04-架构模式

解决特定问题的设计决定的集合，但是可以参数化配置。

  • DSSA 领域特定软件架构

    1. 面向特定任务(task/domain)
    2. 在这个领域里是通用的
    3. 标准结构

  • 架构模式

    1. 把设计决定打包在一起

  • 设计模式

  • 架构模式分类

    1. Module
    2. Component-Connector
      1. 在Runtime动态建立联系
    3. Allocation

Layered Pattern

  • Module类架构
  1. 一个Layer里包含多个Module，不存在跨层的Module
  2. 限制层与层之间的依赖关系，质量属性的可修改性策略就使用了分层
  3. 各层之间不能跨层调用，只能上面调下面

Solution

  • 限制
    1. 每个模块只能被分配到一层
    2. 至少有两层
    3. allowed-to-use关系不能是环，低层不能调用高层
  • 弱点
    1. 增加层很复杂
    2. 性能

Pattern

如果D是sidecar，那么也是分层；但如果D也有一定业务，则不是分层

Broker Pattern

  • Component-Connector类模式

Solution

  • 元素
    1. Client
    2. Server
    3. Broker：中介，帮助client找到server，传递请求
  • 分析
    1. 提升了互操作性、可修改性
    2. 现代场景下如果有多个broker实例，可以提高扩展性、安全性、可用性（补充设计）
    3. 可能单点失效，减低性能、安全性、可用性
    4. 减低可测试性，无法预先确定client和server的状态

Pattern

Model-View-Controller Pattern

从动态看属于Component-Connector
从开发态看属于Module

Solution

Pattern

Pipe-and-Filter Pattern

Component-Connector

Solution

  • 元素

    1. pipe
      • 必须有数据的输入和输出
      • 连接一系列的数据处理/计算
    2. filter
      • 通过pipe连起来

  • 限制

    1. 系统里不可能只有孤立的pipe

Pattern

Client-Server Pattern

Solution

  • 元素

    1. Client
    2. Server

  • 不需要broker：在有限的网络中，有哪些服务是预先知道的

Pattern

Peer-to-Peer Pattern

CC
没有明确的地位差距

Solution

peer即可以发送也可以接受数据

hops：限制可感知的范围
节点可以自由加入退出

  • 分析
    1. 提高了系统的可伸缩性
    2. 牺牲移动安全性

Pattern

Service-Oriented/SOA Pattern

cc

Solution

  • Component
    1. Service Provider
    2. Service Consumer
    3. Registry：注册服务
    4. ESB：client到server的通信路由，类似broker
    5. Orchestration server：编排组织业务
  • Connector:
    1. SOAP
    2. REST
    3. Asynchronous

可以访问到其他组织，不同技术实现的服务

质量属性：提高互操作性

Pattern

虚线：组织边界，内部可以完全由组织控制

Publish-Subscribe Pattern

Solution

存在多对多的关系
不能保证一定会接收到，或即时接收到

Pattern

在OS中事件处理普遍使用

Shared-Data Pattern

CC

Solution

存在中心的被共享的数据

  • 分析
    1. 星型结构，可能单点失效
    2. 想要获取数据高一致性、实时性时的妥协

Pattern

Map-Reduce Pattern

Allocation

Solution

Map：任务分割，一组相同的任务并行，每一个任务都在一个Map上处理
Reduce：把Map的输出整合在一起，变成单一输出

要求数据是大量的，否则map-reduce的代价不划算
通过并行提高了性能

Pattern

Multi-Tier Pattern

Solution

Tier：逻辑的分层，不是真实存在的层。软件的每一个计算资源作为一个Tier

Pattern

Pattern和Tactic

Pattern：一组设计决定
Tactic：最小单元
